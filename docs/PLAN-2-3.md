# PLAN-2-3: protected transport (TLS default + controlled no-TLS fallback)

Источник: `ISSUE.md` (remaining item 2), `docs/rfcs/0002.md` (Фаза 2, пункт 3).

## Цель

Включить защищённый транспорт с обязательной проверкой сертификатов по умолчанию,
с временным явным fallback для dev/test через `--insecure-no-tls`.

## Допущения

- Первая итерация: TLS server + TLS client с verify chain и verify hostname/SAN.
- mTLS остаётся целевым следующим шагом, но не блокирует первый релиз TLS.
- Без валидных сертификатов запуск в TLS-режиме должен завершаться явной ошибкой.

## План работ

## Этап 1. Контракт конфигурации и CLI

Задачи:

1. Добавить параметры транспорта:
   - `--tls-cert`
   - `--tls-key`
   - `--tls-ca`
   - `--tls-server-name` (опционально для валидации peer имени)
   - `--insecure-no-tls`
2. Поддержать загрузку через CLI/ENV/config file с текущим приоритетом.
3. Запретить silent fallback: если TLS-режим активен и параметры невалидны — fail fast.

Проверка:

1. Unit-тесты разбора конфигурации TLS.
2. Тесты ошибок на отсутствующие/битые cert/key/ca.

## Этап 2. Серверный TLS-транспорт

Задачи:

1. Перевести входящий HTTP listener на TLS listener.
2. Проверять корректность пары cert/key на старте.
3. Возвращать явные ошибки и прекращать запуск при проблемах TLS-конфига.

Проверка:

1. Интеграционный тест успешного TLS старта.
2. Тест отказа при невалидной паре cert/key.

## Этап 3. Клиентский TLS-транспорт

Задачи:

1. Перевести исходящие запросы `request_public_key` на HTTPS.
2. Включить обязательную проверку цепочки доверия и SAN/hostname.
3. В режиме `--insecure-no-tls` использовать текущий HTTP-путь с заметным warning.

Проверка:

1. Тест успешного запроса при валидном cert/ca.
2. Тесты отказа: unknown CA, expired cert, SAN mismatch.

## Этап 4. README и операционный runbook

Задачи:

1. Обновить `README.md`:
   - добавить пошаговую генерацию сертификатов (локальный CA + cert/key для нод);
   - добавить примеры SAN (DNS/IP) и требования к именам;
   - добавить примеры запуска `ssh-key-sync start/sync` в TLS-режиме;
   - добавить отдельный блок про `--insecure-no-tls` как dev/test-only.
2. Обновить `docs/man.md` новыми TLS-параметрами и предупреждениями.

Проверка:

1. Инструкции README воспроизводимы в чистом окружении.
2. Команды примеров соответствуют фактическому CLI.

## Этап 5. Финализация

Задачи:

1. Прогнать обязательные проверки через Makefile.
2. Обновить `CHANGELOG.md`.
3. Обновить статус RFC-0002 по результатам реализации пункта 3.

Проверка:

1. `make fmt`
2. `make lint`
3. `make test`
4. `make build`

## Критерии готовности

- TLS включён по умолчанию и требует валидных сертификатов.
- Соединения без валидной проверки сертификата отклоняются.
- Режим `--insecure-no-tls` доступен только по явному флагу и помечен как небезопасный.
- README содержит рабочую инструкцию генерации сертификатов и запуска в TLS-режиме.
