## Problem

В текущей реализации есть набор критичных и системных слабых мест:

1. Ошибка идентичности отправителя в runtime-потоке запроса публичного ключа.
2. Небезопасное хранение управляющих артефактов демона в `/tmp`.
3. Отсутствие защищённого транспорта по умолчанию.
4. Недостаточно жёсткая модель доверия между участниками.
5. Слишком широкие дефолты discovery (UDP broadcast).
6. Кастомный HTTP-разбор вместо промышленного стека.
7. Операционные пробелы в CI/релизной дисциплине и security-проверках.

## Goal

Повысить безопасность, предсказуемость и эксплуатационную надёжность `ssh-key-sync`
поэтапным планом исправлений, который можно выполнить без резкой смены архитектуры
в один шаг.

## Design

Предлагается 3-фазный rollout.

### Фаза 1 (немедленно)

1. [ ] Исправить sender identity в `request_public_key`:
   - всегда подписывать запрос локальным `participant_id`;
   - добавить регрессионный тест на корректный `sender_id`.
2. [x] Убрать daemon-control артефакты из `/tmp`:
   - использовать `$XDG_RUNTIME_DIR/ssh-key-sync/<sid>/`;
   - fallback: `~/.local/run/ssh-key-sync/<sid>/`;
   - права: директории `0700`, файлы `0600`;
   - использовать безопасные флаги создания файлов.

### Фаза 2 (security release)

3. Добавить защищённый транспорт:
   - минимум TLS с обязательной проверкой сертификата;
   - целевое состояние: mTLS для взаимной аутентификации.
4. Ужесточить модель доверия:
   - allowlist участников (`participant_id` + cert/fingerprint binding);
   - отклонять announcements/keys от узлов вне allowlist даже при валидном `SID_TOKEN`.
5. Ограничить discovery по умолчанию:
   - отключить UDP broadcast по умолчанию либо включать только через `--enable-broadcast`;
   - в production предпочитать явный bootstrap/service discovery.

### Фаза 3 (стабилизация)

6. Заменить custom HTTP parsing на промышленный стек (`hyper/axum`):
   - строгий парсинг;
   - лимиты тела запроса;
   - таймауты на уровне библиотечных примитивов.
7. Закрыть операционные пробелы:
   - CI workflow: `fmt`, `lint`, `test`, `build`, `e2e`;
   - синхронизация версии `docs/man.md` с `Cargo.toml`;
   - security scanning зависимостей (`cargo-audit`, policy checks).

## Alternatives

1. Исправлять только критичный пункт №1 и отложить остальное:
   - снижает скорость улучшений и оставляет системные риски.
2. Выполнить все изменения одним релизом:
   - повышает риск регрессий и усложняет откат.
3. Оставить discovery и trust-model без изменений:
   - неприемлемо для production-сценариев.

## Voting

- Решение предлагается к принятию в статусе `draft` → `review` после обсуждения
  объёма Фазы 2 и критериев готовности.

## Migration

Миграция выполняется по фазам без форсированного одномоментного перевода:

1. Фаза 1 — обязательна и блокирует последующие релизы.
2. Фаза 2 — выпускается как отдельный security-релиз с обратной совместимостью
   там, где это возможно.
3. Фаза 3 — стабилизационные изменения с приоритетом на снижение
   операционного и технического долга.
